## ZTS0104

> Re-export source resolves to the module itself (self-cycle)

**카테고리**: 번들러: import/export

### 발생 조건

`export { x } from 'y'` 형태의 named re-export에서 `'y'`가 resolver alias 또는 plugin `onResolve`를 거쳐 **importer 자기 자신**으로 resolve되는 경우.

ZTS의 `__esm` 래퍼는 lazy `__export` getter를 생성하는데, source가 자기 자신이면 getter가 `exports_self.x` 형태로 자기 객체를 다시 참조한다. 호출 시 `RangeError: Maximum call stack size exceeded`로 무한 재귀 발생. rolldown `CIRCULAR_REEXPORT` / esbuild `Detected cycle while resolving import`와 동일한 케이스.

### 잘못된 예

```js
// AssetRegistry.js
export { registerAsset } from '@react-native/assets-registry/registry';
```

```js
// 빌드 옵션
{
  alias: {
    '@react-native/assets-registry/registry': 'react-native/Libraries/Image/AssetRegistry',
  },
}
```

alias가 두 경로를 같은 모듈(`AssetRegistry.js`)로 묶었지만, `AssetRegistry.js` 본인이 그 경로를 import하므로 self-cycle.

### 해결 방법

**1. resolver alias 점검** — alias 결과가 importer 자기 자신으로 가지 않는지 확인. 두 import 경로를 별개 모듈로 두는 게 표준.

```js
// 수정: alias 제거
{
  alias: {
    // '@react-native/assets-registry/registry'를 alias하지 않음
  },
}
```

**2. plugin `onResolve` 점검** — `onResolve` hook에서 source path를 importer path와 비교해 같으면 다른 처리.

**3. 코드에서 자기 자신 import 제거** — 의도한 것이 아니라면 source specifier를 다른 모듈로 변경.

### 참고

- Metro/rspack은 lazy `_$$_REQUIRE` + 모듈 캐시로 self-cycle을 런타임에 우회하지만, ZTS·rolldown·esbuild 같은 ESM 래퍼 출력 모델은 정적 분석에서 거부.
- `export default X` 형태(default re-export)는 별도 경로에서 안전 처리되므로 본 진단 대상이 아님.
