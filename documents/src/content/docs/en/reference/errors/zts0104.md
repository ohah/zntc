## ZTS0104

> Re-export source resolves to the module itself (self-cycle)

**Category**: Bundler — import/export

### When it occurs

A named re-export of the form `export { x } from 'y'` where `'y'` resolves — via resolver alias or a plugin `onResolve` hook — back to the **importer module itself**.

ZTS's `__esm` wrapper emits lazy `__export` getters. When the source module is the same as the importer, the getter becomes `exports_self.x` and references its own enclosing object. Calling it triggers `RangeError: Maximum call stack size exceeded` (infinite recursion). This matches rolldown's `CIRCULAR_REEXPORT` and esbuild's `Detected cycle while resolving import`.

### Bad example

```js
// AssetRegistry.js
export { registerAsset } from '@react-native/assets-registry/registry';
```

```js
// build options
{
  alias: {
    '@react-native/assets-registry/registry': 'react-native/Libraries/Image/AssetRegistry',
  },
}
```

The alias collapses both paths into the same module (`AssetRegistry.js`), but `AssetRegistry.js` itself imports that path — producing a self-cycle.

### How to fix

**1. Audit your resolver alias.** Make sure the alias result does not point back to the importer. Keeping the two import paths as distinct modules is the standard approach.

```js
// Fix: drop the alias
{
  alias: {
    // Do not alias '@react-native/assets-registry/registry'
  },
}
```

**2. Audit your plugin `onResolve` hook.** Compare the source path with the importer path; if equal, return a different resolution (or fall through).

**3. Remove the self-import in source code.** If the self-reference was unintentional, point the import at a different module.

### Notes

- Metro and rspack work around self-cycles at runtime via lazy `_$$_REQUIRE` and the CommonJS module cache. ZTS, rolldown, and esbuild use ESM-wrapper output and reject self-cycles at static analysis time.
- `export default X` (default re-export) is handled through a separate code path and is not subject to this diagnostic.
